# AI 问诊电子病历生成系统商业化技术文档

版本：V1.0  
日期：2026-06-08  
适用范围：门诊问诊、基层医疗、专科门诊、互联网医院、体检中心、诊所集团

## 1. 项目概述

AI 问诊电子病历生成系统是一套面向医疗机构的智能病历辅助系统。系统通过实时录音、语音识别、大模型医学语义分析和结构化病历生成，辅助医生快速完成门诊电子病历书写。

系统核心目标不是替代医生诊疗，而是降低医生病历书写负担，提高问诊记录完整性，提升门诊效率，并为医疗质控、随访、科研数据沉淀提供基础。

当前系统已具备以下核心能力：

- 上传录音转写
- 实时录音分段转写
- 本地 Whisper 语音识别
- DeepSeek 大模型病历生成
- 方言/口音文本标准化
- 医学术语纠错
- AI 病历草稿生成
- 医生确认归档
- TXT / JSON 导出
- 浏览器直接打印病历
- Docker 化部署基础

## 3. 系统功能设计

### 3.1 门诊问诊工作台

医生在工作台中完成完整问诊流程：

```text
选择患者
  ↓
上传录音 / 实时录音
  ↓
语音转写
  ↓
AI 生成病历草稿
  ↓
医生编辑确认
  ↓
导出 / 打印 / 同步 EMR
```

当前页面包含：

- 患者信息区
- 录音上传区
- 实时录音按钮
- 转写文本框
- AI 病历草稿编辑区
- 质控提醒区
- 原文依据区
- 导出和打印功能

### 3.2 语音识别

当前采用本地 `faster-whisper` 作为 ASR 引擎。

能力：

- 支持 MP3、WAV、M4A、WebM 等常见音频格式
- 支持上传录音转写
- 支持实时录音分段转写
- 支持普通话、口音、部分方言和中英混说自动识别
- 支持医学热词提示
- 支持常见医学错词纠正
- 支持重复幻听和乱码过滤

当前配置：

```env
ASR_PROVIDER=whisper
WHISPER_MODEL_SIZE=small
WHISPER_DEVICE=cpu
WHISPER_COMPUTE_TYPE=int8
WHISPER_LANGUAGE=
WHISPER_BEAM_SIZE=8
WHISPER_TEMPERATURE=0
```

商业部署建议：

- 小规模试点：CPU + small/base 模型
- 正式部署：GPU + small/medium 模型
- 高并发部署：独立 ASR 推理服务 + 队列调度

### 3.3 大模型病历生成

当前系统已接入 DeepSeek。

配置：

```env
LLM_PROVIDER=deepseek
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-v4-flash
DEEPSEEK_API_KEY=******
```

处理链路：

```text
原始转写文本
  ↓
方言/口音/错词标准化
  ↓
医学语义理解
  ↓
结构化病历 JSON
  ↓
医生审核编辑
```

生成字段包括：

- 主诉
- 现病史
- 既往史
- 过敏史
- 体格检查
- 初步诊断
- 处理意见
- 缺失项提醒
- 风险提示
- 原文依据

系统约束：

- 未提及的信息必须写“未提及”
- 不允许编造查体和检查结果
- 否定信息必须来自明确表达
- 诊断必须标记为初步考虑或待医生确认
- 医生确认前不得作为正式病历入库

### 3.4 方言与口音适配

系统已支持基础方言和口语标准化。

示例：

| 原始表达 | 标准化结果 |
|---|---|
| 肚屙 | 腹泻 |
| 肚痛 | 腹痛 |
| 心口痛 | 胸痛 |
| 喉咙痛 | 咽痛 |
| 流鼻水 | 流涕 |
| 喘不过气 | 呼吸困难 |
| 冇 | 没有 |
| 唔 | 不 |

后续可按地区扩展：

- 粤语
- 四川话
- 闽南语
- 吴语
- 东北话
- 河南话
- 陕西话

### 3.5 病历打印

医生确认病历后，可直接点击“打印病历”。

打印内容包括：

- 患者姓名
- 性别
- 年龄
- 科室
- 就诊号
- 主诉
- 现病史
- 既往史
- 过敏史
- 体格检查
- 初步诊断
- 处理意见
- 医生签名栏

打印时不会打印工作台、按钮、转写框和质控侧栏。浏览器打印窗口可选择实体打印机或另存 PDF。

## 7. 安全与合规

医疗数据属于敏感数据，商业化必须加强安全设计。

### 7.1 必做安全能力

- HTTPS
- 登录认证
- RBAC 权限控制
- 患者数据加密
- 音频文件加密存储
- API key 环境变量管理
- 审计日志
- 操作留痕
- 医生确认机制
- 数据备份和恢复

### 7.2 AI 安全边界

系统必须明确：

- AI 只生成病历草稿
- AI 不替代医生诊断
- AI 不自动开具处方
- AI 不自动提交正式 EMR
- 医生确认后才可归档或同步

### 7.3 数据出境和第三方模型

如果使用 DeepSeek 云 API，需要根据客户场景确认：

- 是否允许患者数据出医院内网
- 是否需要脱敏后调用
- 是否需要私有化大模型
- 是否符合医院安全评审要求

高等级医院或真实患者数据场景，建议：

- 本地 ASR
- 本地存储
- 私有化大模型
- 院内网部署

## 16. 详细技术优化方向

本章节用于指导系统从原型能力逐步演进为可商业化、可部署、可维护、可规模化交付的医疗 AI 产品。

### 16.1 语音识别优化

当前系统已使用本地 `faster-whisper` 进行真实音频转写，但在真实门诊环境中，识别准确率会受到以下因素影响：

- 医生和患者距离麦克风不同
- 患者声音较小
- 诊室存在背景噪声
- 医患同时说话
- 方言和口音较重
- 药名、检查名、疾病名较复杂
- 实时录音片段过短
- 静音检测误判

#### 16.1.1 模型大小优化

Whisper 模型可按不同业务场景选择：

| 模型 | 速度 | 准确率 | 适用场景 |
|---|---|---|---|
| tiny | 极快 | 较低 | 演示、低配置设备 |
| base | 快 | 一般 | 小诊所、低成本部署 |
| small | 平衡 | 较好 | 当前推荐默认配置 |
| medium | 慢 | 高 | 医院正式试点 |
| large | 很慢 | 更高 | GPU 服务器、高精度场景 |

推荐策略：

- PoC 阶段：`small + CPU`
- 试点阶段：`small + GPU`
- 正式生产：`medium + GPU`
- 高并发部署：独立 ASR Worker 集群

#### 16.1.2 实时转写分段策略

当前系统每 12 秒生成一个音频片段并上传转写。分段时长对实时性和准确率影响较大：

| 分段时长 | 优点 | 缺点 |
|---|---|---|
| 3 秒 | 延迟低 | 上下文少，准确率下降 |
| 5 秒 | 较实时 | 短句容易被切断 |
| 10-12 秒 | 准确率较好 | 延迟略高 |
| 20 秒以上 | 上下文完整 | 实时体验差 |

当前建议：

```text
门诊问诊：10-12 秒
急诊快速记录：5-8 秒
长录音上传：整段转写
```

后续可优化为：

```text
前端音频流
  ↓
VAD 判断有无语音
  ↓
动态切片
  ↓
ASR Worker
  ↓
WebSocket 推送转写
```

#### 16.1.3 VAD 静音检测优化

当前系统使用 Whisper 自带 VAD，并加入兜底重识别策略：

```text
第一遍：开启 VAD，过滤静音
如果文本过短：
第二遍：关闭 VAD，重新识别
```

该策略可以减少低音量患者回答被过滤的问题。

进一步优化方向：

- 使用独立 VAD 模型
- 根据音量动态调整阈值
- 对医生和患者声道分别检测
- 长时间静音不上传
- 停止录音时强制提交最后片段

#### 16.1.4 医学热词优化

系统已加入初始医学热词提示。商业化后应维护可配置医学词库。

词库类型：

- 症状词库
- 疾病词库
- 药品词库
- 检查检验词库
- 科室专科词库
- 方言口语词库

示例：

```text
布洛芬、对乙酰氨基酚、头孢、阿莫西林、奥司他韦
CRP、C反应蛋白、血常规、胸片、CT、心电图
支原体、流感、肺炎、支气管炎、胃炎
```

未来可支持后台配置：

```text
医院词库
  ↓
科室词库
  ↓
医生个人常用词
```

### 16.2 说话人分离优化

当前系统尚未稳定区分医生和患者。商业化部署中，说话人分离非常重要。

#### 16.2.1 方案一：双麦克风

医生和患者分别使用不同麦克风或声道。

优点：

- 准确率最高
- 成本可控
- 实现简单

缺点：

- 需要硬件配合
- 诊室改造成本增加

适合：

- 医院固定诊室
- 高价值专科门诊
- 标准化试点

#### 16.2.2 方案二：声纹分离

使用 diarization 模型区分说话人。

技术链路：

```text
原始音频
  ↓
说话人分离模型
  ↓
speaker_0 / speaker_1
  ↓
ASR 转写
  ↓
DeepSeek 判断医生/患者角色
```

优点：

- 不强依赖硬件
- 可适应已有录音设备

缺点：

- 模型复杂
- 嘈杂环境准确率不稳定
- 多人同时说话效果较差

#### 16.2.3 方案三：大模型语义角色识别

根据话术判断角色。

例如：

```text
“哪里不舒服？” -> 医生
“我头晕好几个月了” -> 患者
```

优点：

- 实现快
- 不增加音频模型复杂度

缺点：

- 只能基于文本判断
- 对短句和上下文依赖较强

推荐商业化组合：

```text
短期：DeepSeek 语义角色识别
中期：双麦克风可选
长期：声纹分离 + 语义纠错
```

### 16.3 大模型病历生成优化

当前系统已经接入 DeepSeek，大模型链路为：

```text
转写文本
  ↓
方言/错词标准化
  ↓
结构化病历生成
```

商业化后建议升级为多阶段链路：

```text
ASR 原始转写
  ↓
对话清洗
  ↓
医生/患者角色识别
  ↓
医学实体抽取
  ↓
病历结构化 JSON
  ↓
质控检查
  ↓
病历文本生成
```

#### 16.3.1 多阶段模型调用

不要让大模型一次完成所有任务。建议拆成：

| 阶段 | 输入 | 输出 |
|---|---|---|
| 对话清洗 | 原始转写 | 标准化问诊文本 |
| 角色识别 | 标准化文本 | 医生/患者对话 |
| 信息抽取 | 对话文本 | 病历字段 JSON |
| 质控检查 | 字段 JSON | 缺失项和风险提示 |
| 文本生成 | 字段 JSON | 病历草稿 |

好处：

- 便于调试
- 降低幻觉
- 支持字段级溯源
- 可单独评估每个阶段

#### 16.3.2 字段级证据溯源

正式商用版本应做到每个字段都有 evidence：

```json
{
  "field": "allergy_history",
  "value": "青霉素过敏",
  "evidence": "患者：我对青霉素过敏。"
}
```

医生点击字段时，可看到原文依据。

价值：

- 增强医生信任
- 便于质控审核
- 降低法律风险
- 支持 AI 结果追责

#### 16.3.3 防幻觉机制

医疗场景必须严格控制幻觉。

防护策略：

- Prompt 明确禁止编造
- JSON Schema 限制输出字段
- 未提及字段写“未提及”
- 诊断必须写“待医生确认”
- 查体和检查结果不得自动生成
- 输出后进行规则校验
- 医生确认前不得入正式 EMR

#### 16.3.4 模型失败回退

当前系统已支持 DeepSeek 失败回退规则版。

商业化建议：

```text
DeepSeek 调用
  ↓ 失败
重试一次
  ↓ 仍失败
规则版生成
  ↓
提示医生当前为降级结果
```

### 16.4 实时交互优化

当前实时转写属于“分段准实时”。正式产品可继续升级。

#### 16.4.1 WebSocket 实时推送

推荐架构：

```text
浏览器 MediaRecorder
  ↓ WebSocket
API 网关
  ↓
ASR Worker
  ↓
WebSocket 推送文本
```

优势：

- 更低延迟
- 更自然的实时体验
- 支持识别中状态
- 支持分段更新和最终修正

#### 16.4.2 转写稳定化

实时 ASR 经常出现前后片段重复。

优化策略：

- 相邻片段文本相似度去重
- 最后一段和下一段重叠比对
- 标记临时转写和最终转写
- 停止录音后统一做最终整理

#### 16.4.3 自动生成病历时机

可支持三种模式：

| 模式 | 说明 |
|---|---|
| 手动生成 | 医生点击生成草稿 |
| 录音停止后自动生成 | 停止问诊后自动生成 |
| 边问边生成 | 实时更新结构化字段 |

商业化建议先采用：

```text
录音停止后自动生成 + 医生手动刷新
```

## 18. 生产级系统架构演进

### 18.1 当前原型架构问题

当前系统适合演示和试点，但存在以下生产化不足：

- SQLite 不适合多用户并发
- ASR 和 API 在同一进程
- 缺少登录权限
- 缺少完整审计页面
- 缺少任务队列
- 缺少 WebSocket
- 缺少文件生命周期管理
- 缺少 HIS / EMR 集成

### 18.2 推荐生产架构

```text
前端 Web
  ↓
Nginx / HTTPS
  ↓
API 服务
  ├─ 用户权限服务
  ├─ 患者就诊服务
  ├─ 病历服务
  └─ 集成服务
  ↓
Redis / Kafka
  ↓
ASR Worker 集群
  ↓
LLM Provider
  ↓
PostgreSQL + MinIO
```

### 18.3 服务拆分建议

| 服务 | 职责 |
|---|---|
| Gateway | 鉴权、限流、路由 |
| User Service | 用户、角色、权限 |
| Visit Service | 患者和就诊记录 |
| Audio Service | 音频上传、存储、清理 |
| ASR Service | 语音识别 |
| LLM Service | 病历生成 |
| EMR Service | 病历草稿、确认、导出 |
| Audit Service | 审计日志 |
| Integration Service | HIS / EMR 对接 |

### 18.4 队列化处理

长音频转写不应阻塞 HTTP 请求。

推荐：

```text
POST 上传音频
  ↓
创建任务
  ↓
Redis/Celery 队列
  ↓
ASR Worker 执行
  ↓
前端轮询或 WebSocket 获取结果
```

## 19. HIS / EMR 对接方案

### 19.1 对接目标

第一阶段建议只做两个方向：

```text
从 HIS 拉取患者和就诊信息
向 EMR 推送医生确认后的病历
```

不建议初期开放：

- 自动诊断写入
- 自动处方写入
- 自动医嘱写入

### 19.2 对接方式

| 方式 | 说明 |
|---|---|
| REST API | 现代系统推荐 |
| HL7 v2 | 老医院系统常见 |
| FHIR | 标准化程度高 |
| 数据库视图 | 快速但耦合高 |
| 文件交换 | 过渡方案 |

### 19.3 同步策略

```text
AI 草稿
  ↓
医生编辑
  ↓
医生确认
  ↓
生成最终病历
  ↓
同步 EMR
  ↓
记录同步日志
```

同步失败时：

- 保留本地病历
- 标记 `sync_failed`
- 支持手动重试
- 记录错误日志

## 20. 安全合规详细方案

### 20.1 权限控制

建议采用 RBAC：

```text
用户
  ↓
角色
  ↓
权限
  ↓
资源
```

角色：

- 医生
- 护士
- 科主任
- 管理员
- 审计员

### 20.2 数据加密

需要加密的数据：

- 患者姓名
- 手机号
- 身份证号
- 地址
- 音频文件
- 病历内容
- API key

建议：

- 数据库透明加密
- 字段级加密
- 对象存储服务端加密
- HTTPS 传输加密
- 密钥管理服务

### 20.3 审计日志

必须记录：

- 登录
- 查看患者
- 上传录音
- 生成病历
- 修改病历
- 确认病历
- 导出病历
- 打印病历
- 同步 EMR

审计字段：

```json
{
  "user_id": "D001",
  "action": "confirm_emr",
  "resource_type": "emr_record",
  "resource_id": "R001",
  "ip": "192.168.1.10",
  "created_at": "2026-06-08T10:00:00"
}
```

### 20.4 模型调用合规

如果调用云端大模型，需要进行：

- 数据分类分级
- 患者信息脱敏
- 合同和数据处理协议
- 安全评估
- 日志留存
- 可追溯性设计

若客户要求数据不出院：

- 本地 ASR
- 私有化 LLM
- 内网部署
- 本地对象存储
